# 4단계 — 표지 이미지 소싱 설계 (`photos.py` + `backgrounds.py` 개편)

- 상위 스펙: `docs/superpowers/specs/2026-07-16-card-redesign-design.md` §5.4
- 브랜치: `card-redesign` (1·2·3단계a 완료 위에 얹음)
- 날짜: 2026-07-16

## 1. 목표

표지에 **실제 이미지**를 최대한 채운다. 순수 그래픽은 최후의 수단이지 기본값이 아니다.
북극성은 조회수·팔로워이고, 스펙이 표지를 "최대 성장 레버"로 규정했다.

## 2. 사용자 결정 (이 세션, 2026-07-16)

1. **완전 자동 + 신뢰 점수** — 발행 시점 사람 검수 없음. 사전 큐레이션도 아님.
2. **기사 사진 우선, 신규 유료 API 없음** — 웹 이미지 검색 API(Google CSE·Bing)는 붙이지 않는다.
3. **얼굴 검출 없음 (지금은)** — 고정 크롭으로 4단계를 닫고, 실제 표지를 눈으로 본 뒤 재판단.
4. **인물 라이브러리는 폴백으로 강등** — 콜라주 폐기, 확대하지 않음.
5. **Claude 비전으로 후보 판정** — 사물컷 배제·최고 컷 선택.
6. **뉴스 사진 출처 표기 안 함** — 단, CC BY 크레딧은 유지(§7).

## 3. 실측 (2026-07-16, 직접 측정함)

설계의 근거. **추측 아님.** 스크립트는 일회성이라 저장소에 없다.

| 매체 | RSS 내장 이미지 태그 | 기사 og:image |
|---|---|---|
| 연합뉴스 | **있음** — `media:content`, 기사당 최대 4장, 실사진 | 있음 (`_P4` = 더 큰 버전) |
| 매일경제 | **있음** — `media:content` | 있음, 단 일부는 플레이스홀더 |
| WSJ | **있음** — `media:content` (`images.wsj.net/im-...`) | **페이지 403** |
| 한국경제 | 없음 | **로고 플레이스홀더뿐** |
| The Economist | 없음 | **페이지 403** |

여기서 나온 세 가지 사실이 설계를 지배한다:

- **RSS 내장 `media:content`가 og:image보다 낫다.** 기사 페이지를 안 가도 되고(WSJ은 페이지가 403인데
  RSS엔 이미지가 그대로 있다), 빠르고, 봇 차단을 안 당하고, 기사당 여러 장이 온다.
  → **기사 페이지는 가져오지 않는다.**
- **플레이스홀더 함정이 실재한다.** 한경 og:image는 전부 `static.hankyung.com/img/logo/logo-news-sns.png`
  (= 한경 로고). 매경도 이미지 없는 기사는 `static.mk.co.kr/facebook_mknews.jpg`를 준다.
  **og:image가 "있다"고 다 사진이 아니다.** 안 거르면 표지에 한국경제 로고가 대문짝만하게 나간다.
  (죽은 WSJ 피드가 200을 주던 것과 같은 종류의 함정.)
- **같은 사진이 매체를 건너 나타난다.** 매경이 연합 사진을 `rcv.YNA.20260716.PYH2026071617330001300_R.jpg`로
  받아쓴다 — 파일명에 연합 사진 ID(`PYH2026071617330001300`)가 박혀 있고, 연합 원본은 같은 ID의 `_P2.jpg`다.
  → **ID로 매칭하면 크로스소스 빈도가 잡힌다. 여러 매체가 고른 사진 = 그 이슈의 대표 사진.**

**한경·Economist는 이미지가 구조적으로 0**이다(한경은 `description`도 없다 — 같은 결). 버그가 아니라
그 매체의 성질이다. 폴백 체인이 받는다.

## 4. 아키텍처

```
collector.py    RSS/Atom → Article(+ images: list[str])      ← 신규 필드
     ↓
issues.py       Article[] → Issue[]  (크로스소스 랭킹 — 3단계a 완료, 수정 없음)
     ↓
photos.py       Issue → 후보 수집 → 기계 필터 → Claude 비전 선택   ← 신규 모듈
     ↓
backgrounds.py  build_background(): 기사 사진 → 인물 → 위키미디어 → Unsplash → None
     ↓
renderer.py     render(background=...) → render_cover(background=...)  ← render()에 통로 추가
```

**`renderer.render()`에 배경 통로가 없다 (코드 확인함).** 814행이 `render_cover(briefing.headline,
when, fonts, theme=theme)` — `background`를 안 넘긴다. `render_cover` 자체는 2단계에서 사진 경로를
지원하지만, **데일리 브리핑은 그 인자를 받은 적이 없다.** 그래서 데일리 표지는 여태 늘 그래픽이었고,
스펙이 표지를 "최대 성장 레버"로 부른 이유가 이것이다. `render()`에 `background: Background | None = None`을
추가해 `render_cover`로 넘긴다.

참고: `renderer.render()`의 현재 호출부는 `tests/test_renderer.py` **뿐이다.** 데일리 파이프라인을
모듈로 묶는 일(`collect → summarize → render → 호스팅 → 발행`)은 아직 안 된 상태이고
(메모리의 "남은 일"), 4단계 범위가 아니다. 4단계는 `render()`가 배경을 **받을 수 있게** 만드는 데까지다.

**`photos.py`를 새로 만드는 이유**: `backgrounds.py`가 이미 262줄이고 소스별 fetch를 다 이고 있다.
후보 수집·필터·Claude 판정을 얹으면 400줄을 넘고 책임이 뒤섞인다.

- `photos.py` = "이슈에서 최고의 사진 한 장 고르기"
- `backgrounds.py` = "체인 돌려서 배경 하나 내놓기"

경계는 `Issue → Background | None` 하나. 각각 따로 테스트된다.

### 4.1 `collector.py` 변경

```python
@dataclass
class Article:
    source: str
    title: str
    link: str
    published: datetime
    summary: str = ""
    language: str = "ko"
    images: list[str] = field(default_factory=list)   # 신규
```

RSS·Atom 두 생성 지점에서 `media:content` / `media:thumbnail` / `enclosure`의 `@url`을 뽑는다.
네임스페이스가 붙으므로 태그 로컬명으로 비교한다(`tag.split("}")[-1]`).

기본값이 있는 필드라 **기존 `Article(...)` 호출부는 전부 그대로 돌아간다**(3단계a 테스트 포함).

## 5. `photos.py`

### 5.1 후보 수집

이슈의 기사들에서 `images`를 평탄화하고, **URL을 정규화해 같은 사진을 병합하며 빈도를 센다.**

연합 사진 ID(`PYH` + 숫자, `PCM` + 숫자)를 파일명에서 추출해 키로 쓴다. 매경의 `rcv.YNA...` 파일명과
연합 원본이 같은 키로 묶인다. ID가 없으면 URL 자체를 키로 쓴다(병합 안 됨 — 안전한 저하).

```python
@dataclass(frozen=True)
class Candidate:
    url: str
    sources: frozenset[str]   # 이 사진을 실은 매체들
    freq: int                 # 등장 횟수
```

### 5.2 기계 필터 (Claude에 보내기 전)

확실한 쓰레기만 제거한다. 여기서 품질 판정을 하지 않는다 — 그건 Claude 몫이다.

- **플레이스홀더 블랙리스트**: `static.hankyung.com/img/logo/`, `static.mk.co.kr/facebook_`,
  일반 패턴 `/logo/`, `_sns.png`, `facebook_`. (실측으로 확인된 것들.)
- **최소 해상도**: 다운로드 후 Pillow로 확인, **짧은 변 640px 미만 배제**.
  표지가 1080×1350이라 그 미만은 확대하면 뭉갠다.
- **정렬**: 빈도 내림차순 → 해상도 내림차순.
- **상한**: 상위 **6장**만 Claude에 넘긴다(비용 상한).

**연합 P2→P4 승격**: RSS는 `_P2`(작은 것), og:image는 같은 사진의 `_P4`(큰 것)를 준다(실측).
**URL 문자열 치환으로 `_P4`를 만들어 먼저 시도하고, 404면 `_P2`를 쓴다** — 기사 페이지를 가져오는
게 아니다(§3 "기사 페이지는 가져오지 않는다"와 모순 없음). 추측 위에 서지 않도록 실패를 정상
경로로 다룬다.

### 5.3 Claude 비전 선택

후보 6장을 첨부하고 구조화 출력으로 받는다. 모델은 기존과 같은 `claude-sonnet-5`.

```python
{"pick": int | None, "reason": str}
```

프롬프트 기준은 스펙 §5.4 그대로:

- 인물 얼굴 > 기업 로고 > 극적 실사
- **사물 설명 사진 배제** — 사옥 정면, 웨이퍼, 제품컷, 스펙 표가 박힌 전시 보드
- 감정·표정이 강한 컷 선호(놀람·긴장·환호). 밋밋한 증명사진식 회피
- 쓸 게 없으면 `pick: null` — 억지로 고르지 않는다

**이것이 팬아트 사고의 재발이 아닌 이유**: 그때 터진 건 "이 사진이 그 사람이 맞나"를 기계가 판정하지
못해서였다. 지금 후보는 검색으로 찾은 게 아니라 **그 이슈를 다룬 기사에 매체가 직접 붙인 사진**이라
관련성이 편집자에 의해 이미 보장돼 있다. Claude는 신원 확인이 아니라 **이미 관련 있는 N장 중 가장 센 컷
고르기**만 한다.

비용: 실행당 약 +$0.01 (현재 $0.017~0.024) → 월 $1 미만.

## 6. `backgrounds.py` 개편

```
build_background(people, bg_query, *, issue=None, ...) -> Background | None
  1. photos.pick(issue)      기사 사진 + Claude 판정      ← 신규 1순위, issue 있을 때만
  2. 인물 라이브러리 (단일 얼굴 블리드)                    ← 콜라주 폐기
  3. fetch_wikimedia(bg_query)
  4. fetch_unsplash(bg_query)
  5. None → 그래픽 표지 (render_cover variant="dark"|"color")
```

**`issue`는 기존 인자 뒤의 선택 키워드로 넣는다.** 현재 호출부는 `ai_brief.py:391`,
`blog_brief.py:406`, `backgrounds.py:247`(main), 그리고 테스트 2곳인데 **전부 `(people, bg_query)`
위치 인자로 부른다.** `issue`를 앞에 끼우면 다 깨진다. 게다가 **AI 브리핑·블로그 요약에는 `Issue`가
아예 없다** — 이슈 클러스터링은 데일리 경제 브리핑만의 개념이다. `issue=None`이면 1번을 통째로
건너뛰므로 두 모듈은 손대지 않아도 지금 그대로 동작한다.

**3·4 순서를 현재와 바꾼다.** 지금은 Unsplash가 먼저인데 스펙 우선순위에 비추면 거꾸로다.
실측 기록: "**Unsplash는 인물 커버리지가 0이다.** 키를 받아도 늘어나는 건 주제 배경의 미적 품질뿐 —
인물은 공용에서만 온다." 인물·로고가 있는 쪽은 위키미디어다. 게다가 `UNSPLASH_ACCESS_KEY`는 아직
발급 대기 중이라 4번은 현재 항상 건너뛰어진다.

**`compose_people()` 콜라주 폐기** (스펙 §5.4 지시). 인물 라이브러리는 남기되 **단일 얼굴 블리드**로만
쓴다. 라이브러리는 기사에 사진이 없는 이슈(한경 단독 등)의 안전망이다. 확대하지 않는다 —
케빈 워시 같은 현직 인물은 이제 기사 사진이 덮는다.

**2인 이상이면 첫 번째 인물만 쓴다.** 상위 스펙 §5.4는 "2인이면 대비 구도(분할)"를 적었지만,
이 경로는 이제 **폴백의 폴백**이다(기사 사진이 1순위를 가져간 뒤에야 닿는다). 거의 안 쓰일 경로에
분할 합성 렌더러를 새로 짜는 것은 과잉이다. 실제로 2인 표지가 자주 나오는 게 확인되면 그때 만든다
(§11). 모델이 `people`을 우선순위 순으로 주므로 첫 번째가 그 이슈의 주인공이다.

## 7. 크레딧 (라이선스)

**뉴스 사진(연합·매경·WSJ)은 출처를 표기하지 않는다** (사용자 결정).

**단, CC BY 크레딧은 그대로 유지한다.** 이번에 해제된 것은 "신문 사진을 안 쓴다"는 **자체 규칙**이고,
CC BY는 "쓰려면 크레딧을 달라"는 **라이선스 자체의 조건**이라 성격이 다르다. 위키미디어·인물
라이브러리 경로의 `Background.credits`는 손대지 않는다. 빼면 실제 라이선스 위반이다.

DMCA 리스크는 사용자 소유 결정(상위 스펙 §5.4)에 포함된다.

## 8. 에러 처리

**배경 실패는 절대 발행을 막지 않는다** (기존 원칙 유지). 모든 실패를 삼켜 `errors`에 남기고
다음 체인으로 간다.

**Claude 비전 호출이 실패하면 기계 필터 1등을 자동 채택하지 않고 다음 체인으로 내려간다.**
자동 1등 채택은 정확히 팬아트 사고의 경로다. 신뢰 점수를 매기는 주체가 Claude이므로 Claude가
없으면 점수도 없다. 점수 없는 후보를 1등이라 쓰는 건 "자동 1등 선택 금지"를 우회로 어기는 것이다.
API가 죽은 날은 표지가 라이브러리·위키미디어·그래픽으로 나간다 — 못생겨질 뿐 사고는 안 난다.

개별 후보의 다운로드 실패·타임아웃은 그 후보만 건너뛴다. 이미지 CDN 429는 기존 `download()` 백오프
패턴을 재사용한다(`upload.wikimedia.org`에서 실제로 맞은 적 있음).

## 9. 테스트

표준 `unittest`, **네트워크 없이** (pytest 아님 — `python -m unittest discover -s tests -q`).

- `collector`: 가짜 RSS/Atom XML → `images` 추출 (`media:content` 다중 · `enclosure` · 없음)
- `photos`: 연합 사진 ID 매칭으로 크로스소스 병합 / 플레이스홀더 블랙리스트 테이블 /
  해상도 필터 / **가짜 Claude 클라이언트**로 pick·null 경로
- `backgrounds`: 체인 폴백 순서(각 소스 스텁) / 실패 삼킴 / `issue=None`이면 1번 건너뜀
- `renderer`: `render(background=...)`가 표지에 사진 경로를 태우는지
- 회귀: 기존 248개 유지

**기존 테스트 중 바뀌는 것**: `test_backgrounds.py:51~66`의 `compose_people` 테스트 4개는 콜라주
2인 배치를 검증한다. 콜라주를 폐기하므로 단일 얼굴 계약으로 다시 쓴다(삭제가 아니라 대체 —
2인 이슈에서도 얼굴 하나를 고르는 규칙이 필요하다).

## 10. 범위

**포함**: `Article.images` · `photos.py` 신규 · `build_background` 체인 개편 · 콜라주 폐기 ·
`renderer.render()`에 `background` 통로 추가.

**제외** (근거와 함께):

- **`render_article` 제거 — 취소.** 3단계a 원장이 "죽은 코드 됨(render_issue로 대체) — 정리 후보"로
  적었지만 **틀렸다. 살아 있는 코드다** — `blog_brief.py:35`가 import하고 `:149`가 쓴다
  (`summarizer.py:135` 정의, 테스트 `test_summarizer.py:86·92·97`). 데일리 브리핑 흐름에서만 대체된
  것이지 블로그 요약 모듈은 그대로 쓴다. 지웠으면 `blog_brief`가 깨졌다.

- **얼굴 검출** — 뉴스 사진이 실제로 얼마나 잘 크롭되는지 아무도 안 봤다. 실물 보고 판단.
- **기업 로고 전용 소싱** — 제대로 하려면 기업명 정규화 + 로고 자산 큐레이션 + 매칭이 붙는 별도
  하위 프로젝트. 기사 사진이 1순위로 대부분을 덮고, 기사 사진이 로고컷이면 Claude가 그걸 고른다
